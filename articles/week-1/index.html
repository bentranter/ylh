Deploying a Static Site to GitHub Pages with Wintersmith

Or, how this site was built

I figured I’d keep it simple this week, so let’s get a static site up and running on GitHub Pages, with Node.js and Wintersmith. Anytime I use a new or different technology, I’ll do my best to explain how to install it. Let’s get started by installing Node.

Installing Node

Head on over to nodejs.org and click the ‘Install’ button. From there, just run the installer once you’ve downloaded it, and you should be good to go. To test it out, open your console. On a Mac, its just called Terminal and can be found in your applications, or by searching with spotlight, and on Windows, its called Console and can be found from the start menu, or by searching for it (I haven’t used Windows in a while, please correct me if I’m wrong).

You can test to see if Node installed correctly by running:

$ node
Note: you don’t actually type the ‘$’ symbol, its just used to inform you that you should be running a command instead of typing code.

If it works, the Node REPL will start. You can hit Control-C on any system to exit the Node REPL. Congrats! You’ve installed Node.js.

NPM

NPM stands for ‘Nice People Matter’, or ‘Node Package Manager’ (whichever you prefer). Its a package manager for JavaScript. NPM should install automatically when you install Node, but let’s make sure its there. Just run:

$ npm

As long as you don’t get a ‘command not found’ type of error, everything is good. Now, we’ll install Wintersmith using NPM.

Install Wintersmith

This is as easy as running one command:

$ npm install wintersmith -g

Where install means download a package to your computer, -g is a flag that instructs the package to be downloaded globally, so you can use that package anywhere (as opposed to only within a certain project directory), and finally wintersmith is the name of the package we want to install.

Note that installing wintersmith may take an unusually long time to install (like more than five minutes), but that’s okay, nothing’s going wrong so just be patient :)

Generate A New Site

This is as simple as running:

$ wintersmith new project-name

where project-name is obviously the name of your site. From here, enter the directory of your new site and preview it by running:

$ cd project-name
$ wintersmith preview

and open up localhost:8080 in a web browser. You’ll see a website built with the default theme.

Writing A New Post

Writing a new post is as simple as creating a new folder in the articles directory, and creating a new markdown document within the new directory. To do this, we’ll stop the server (type Control-C), and then run a few simple commands:

$ cd contents/articles
$ mkdir new-post
$ cd new-post
$ touch new-post.md

What we’ve done here is change into the directory named articles, created a new folder using the $ mkdir command, navigated to that folder, and created a new markdown document using the $ touch command. From here, you should go to a sample post to copy/paste the metadata (the stuff at the top of the document in between the ---‘s) and replace it with your own info. At this point, you can write your post! Check out this mardown cheatsheet for a handy reference on how to write in markdown if you’re new to it. Start your server again once you’ve written a post to preview it.

Building Your Static Site For Deployment

Wintersmith makes this really easy. Just run:

wintersmith build

This will generate a site made up of static HTML and CSS pages. The site will appear in a new folder called build, located at the top of your new sites directory. This new site is what GitHub requires to host our site.

Deploying to GitHub Pages
Note: You’ll need git installed for this section to be possible. That is outside the scope of this tutorial, but a simple Google search should get you sorted out :)

First, make sure you’ve actually signed up for a GitHub account. Just got to GitHub to do that, it’s really easy. From there, find the ‘+’ at the top beside your name, and click ‘New Repository’.

Name the new repo username.github.io. It must match your username exactly, or this won’t work. Once that’s done, you’ll need to clone the repository locally:

$ git clone https://github.com/username/username.github.io

After doing that, copy the contents from the build folder we generated into this repository (it’ll just appear as a folder called username.github.io somewhere on your computer). We’re almost done now, finish off by comitting the changes:

$ git add --all
$ git commit -m "First commit"
$ git push

And that’s it! Your site will now be live at username.github.io. If you’re wondering what those git commands do, I’ll break it down. The command add -all adds each changed file to be staged. The command commit -m means that you’re comitting the staged changes to the master branch, with the comment ‘First commit’. push simply syncs the changes from your local repository to your remote repository on GitHub.
